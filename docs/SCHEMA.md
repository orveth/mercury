# Mercury Database Schema

Canonical reference for Mercury's SQLite schema. All consumers of the Mercury database should use this document as their source of truth.

**Database location:** `~/.local/share/mercury/mercury.db`

**Engine:** SQLite with WAL mode and `busy_timeout=5000`

## Tables

### messages

The append-only message log. Messages are never deleted by Mercury.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Monotonically increasing message ID. Used by cursors to track read position. |
| `channel` | TEXT | NOT NULL | Destination channel name (e.g. `status`, `workers`, `keeper:murmur`). |
| `sender` | TEXT | NOT NULL | Self-chosen agent identity (e.g. `oracle`, `worker:auth`). |
| `body` | TEXT | NOT NULL | Opaque message text. Mercury never interprets content. |
| `created_at` | TEXT | NOT NULL, DEFAULT `strftime('%Y-%m-%dT%H:%M:%fZ', 'now')` | RFC3339 timestamp with fractional seconds, always UTC. |

**Indexes:**
- `idx_messages_channel` on `(channel)` -- speeds up channel-filtered queries
- `idx_messages_created` on `(created_at)` -- speeds up time-range queries

### subscriptions

Tracks which agents are subscribed to which channels. Used by `read` to determine which channels an agent receives messages from.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `agent` | TEXT | NOT NULL | Agent identity string. |
| `channel` | TEXT | NOT NULL | Channel name. |
| `created_at` | TEXT | NOT NULL, DEFAULT `strftime('%Y-%m-%dT%H:%M:%fZ', 'now')` | When the subscription was created. |

**Primary key:** `(agent, channel)`

### cursors

Tracks each agent's read position per channel. The `last_read_id` points to the highest `messages.id` the agent has seen, so unread messages are those with `id > last_read_id`.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `agent` | TEXT | NOT NULL | Agent identity string. |
| `channel` | TEXT | NOT NULL | Channel name. |
| `last_read_id` | INTEGER | NOT NULL, DEFAULT 0 | Highest message ID this agent has read on this channel. |

**Primary key:** `(agent, channel)`

### routes

*Added in PR #2 (`feat/routes` branch). Not yet on `main`.*

Configures transport routing -- how messages on a channel get forwarded to external destinations (e.g. Discord, webhooks).

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Route identifier. |
| `channel` | TEXT | NOT NULL | Channel to watch. Use `*` for all channels. |
| `destination` | TEXT | NOT NULL | Transport target (e.g. `discord:1234567890`). |
| `config` | TEXT | NOT NULL, DEFAULT `'{}'` | JSON configuration for the transport. |
| `active` | BOOLEAN | NOT NULL, DEFAULT 1 | Whether this route is enabled. |
| `created_at` | TEXT | NOT NULL, DEFAULT `strftime('%Y-%m-%dT%H:%M:%fZ', 'now')` | When the route was created. |

**Unique constraint:** `(channel, destination)`

**Index:** `idx_routes_channel` on `(channel)`

## Relationships

```
messages.id  <--  cursors.last_read_id
     ^                    |
     |                    |
  "unread = messages where id > cursor's last_read_id"
     |                    |
     v                    v
messages.channel  ==  cursors.channel  ==  subscriptions.channel
                      cursors.agent    ==  subscriptions.agent
```

- An agent subscribes to channels via `subscriptions`.
- When reading, Mercury finds all subscribed channels, then returns messages with `id > last_read_id` from the `cursors` table.
- If no cursor exists for an agent+channel pair, all messages on that channel are considered unread (cursor defaults to 0).
- After reading, the cursor is updated to the highest message ID seen.

## Timestamp format

All `created_at` columns use RFC3339 with fractional seconds in UTC:

```
2026-03-22T19:32:45.123Z
```

Generated by SQLite's `strftime('%Y-%m-%dT%H:%M:%fZ', 'now')`. The `%f` format produces seconds with 3 decimal places.

## For implementors

If you are building a new consumer of the Mercury database, here is what you need to know.

### Opening the database

- Path: `~/.local/share/mercury/mercury.db`
- Open as SQLite with WAL journal mode: `PRAGMA journal_mode=WAL`
- Set a busy timeout for concurrent access: `PRAGMA busy_timeout=5000`
- If your consumer is read-only, open with `readonly` mode. The Discord feed does this.

### Reading messages

The simplest query to get new messages since a known position:

```sql
SELECT id, channel, sender, body, created_at
FROM messages
WHERE id > ?
ORDER BY id ASC
LIMIT ?
```

For channel-filtered reads:

```sql
SELECT id, channel, sender, body, created_at
FROM messages
WHERE channel = ? AND id > ?
ORDER BY id ASC
```

### Cursor management

If your consumer tracks its own read position (recommended), you have two options:

1. **Use Mercury's `cursors` table** -- subscribe as an agent and use `INSERT ... ON CONFLICT UPDATE` on the cursors table. This is what the CLI and MCP server do.

2. **Use an external cursor** -- store the last-seen message ID in a file or your own state. This is what the Discord feed does (writes to `~/.local/share/mercury/discord-feed-cursor`).

Option 2 is simpler if you only need to poll `messages` and don't need subscription management.

### Writing messages

```sql
INSERT INTO messages (channel, sender, body) VALUES (?, ?, ?)
```

The `id` and `created_at` are auto-generated. Do not provide them.

### Schema validation

Before operating, validate that the tables you depend on exist with the expected columns. Use `PRAGMA table_info(table_name)` and check that the columns you query are present. This catches version mismatches early with a clear error message.

### Concurrency

Multiple processes can read and write simultaneously thanks to WAL mode. The Go CLI sets `MaxOpenConns=1` implicitly (single `*sql.DB` connection). Consumers that only read can safely open multiple connections.

## Making schema changes

When modifying Mercury's schema:

1. **Update the Go source** -- `internal/db/db.go` contains the `schema` constant with all `CREATE TABLE IF NOT EXISTS` statements. This is the source of truth for migration.
2. **Update this document** -- keep `docs/SCHEMA.md` in sync with the actual schema.
3. **Notify downstream consumers** -- the following consumers depend on the schema:
   - **MCP server plugin** (`damsac-studio/plugins/mercury/server.ts`) -- uses prepared statements against `messages`, `subscriptions`, and `cursors`.
   - **Discord feed tool** (`tools/discord-feed/index.ts`) -- reads from `messages` only.
   - Any future consumers should be listed here.
4. **Consider backwards compatibility** -- all tables use `CREATE TABLE IF NOT EXISTS`, so adding new tables is safe. Adding columns to existing tables requires `ALTER TABLE` migration logic.
